Objective

Add the ability to click on a synced external calendar event (Google, Microsoft, Apple/Yahoo CalDAV) in the visual calendar and mark it as "Free" so it no longer blocks GoStork scheduling. Extend the existing availability architecture rather than building a new system.

Architectural Analysis

Current Override System

AvailabilityOverride model: { id, userId, date, isAvailable, slots (JSON), label } with a @@unique([userId, date]) constraint.
Used for day-level overrides: "on this date, I'm unavailable" or "on this date, I'm only available 9-12".
The availability engine (getAvailability) checks overrides first to determine the day's base time windows, before subtracting external calendar conflicts.
Why a Separate Model Is Better Than Adding a Column to AvailabilityOverride

The existing AvailabilityOverride is date-scoped with a @@unique([userId, date]) constraint — one override per user per date. Event-level free overrides are fundamentally different: there can be many per date, they're tied to a specific external event (not a time window), and they interact at the conflict-checking stage (not the base-window stage). Adding externalEventId to AvailabilityOverride would violate the unique constraint and conflate two different concepts.

Decision: Create a lightweight EventFreeOverride model purpose-built for this feature. It complements the existing override architecture by operating at a different layer: AvailabilityOverride controls the base time windows, while EventFreeOverride controls which external conflicts are ignored within those windows.

Event Identification Strategy

External events have provider-specific IDs that are stable across fetches:

Google: event.id (stable string like abc123def)
Microsoft: event.id (stable Graph API ID)
CalDAV: event.id (UID from iCal data)
On the frontend, events are prefixed (goog-, msft-, caldav-) to avoid collisions. The override table will store the raw provider event ID (without prefix) plus the provider field for disambiguation.

Availability Engine Integration

Currently, getAvailableSlots builds a googleBusyIntervals array from all providers' freeBusy responses, then checks each slot against it. The fix: enhance the freeBusy data flow to carry event IDs alongside time ranges. Before adding a busy interval, check if its event has a free override → if so, skip it.

Google FreeBusy API limitation: Google's freebusy.query doesn't return individual event IDs — only aggregated busy blocks. To support per-event overrides for Google, we'll switch the conflict-checking to use getEvents (which does return event IDs) instead of getFreeBusy for calendars where the user has active overrides. For Microsoft and CalDAV, the existing getFreeBusy methods already enumerate individual events with IDs, so filtering is straightforward.

Add new model to prisma/schema.prisma:
model EventFreeOverride {
  id              String   @id @default(uuid())
  userId          String
  user            User     @relation(fields: [userId], references: [id])
  externalEventId String   // raw provider event ID
  provider        String   // "google" | "microsoft" | "apple" | "yahoo"
  calendarId      String   // which calendar the event belongs to
  title           String?  // snapshot of event title for display
  createdAt       DateTime @default(now())
  @@unique([userId, externalEventId, provider])
}
