Objective:
Build a flexible, single-component dynamic "Costs" system for GoStork providers (Surrogacy Agencies, Egg Donors, IVF Clinics) with AI-powered PDF/Excel parsing, an admin approval workflow, and full version history. Do not duplicate code for different provider types; build a single modular component that adapts based on the provider's providerType and mandatory fields.

1. Database Schema Updates (Prisma):
Create or update the schema to support dynamic costs, versioning, and templates.

CostTemplate: Admin-managed table storing mandatory fields per providerType. Fields should include id, providerType, fieldName, fieldDescription, isMandatory.

ProviderCostSheet: Represents a specific cost sheet submission. Fields: id, providerId, fileUrl (Supabase Storage URL), status (PENDING, APPROVED, REJECTED), adminFeedback (text), version (integer, increments on new approvals), createdAt, updatedAt.

CostItem: The actual line items linked to a ProviderCostSheet. Fields: id, providerCostSheetId, key (name of the cost), value (amount/range), isCustom (boolean - true if generated by AI and not in the GoStork template), comment (optional notes).

2. Backend (NestJS):

Storage Integration: Create an endpoint to upload PDF/Excel cost sheets directly to a dedicated Supabase Storage bucket (provider-cost-sheets).

AI Parsing Service: Utilize the already installed @google/genai or @google/generative-ai package. Create a service that takes the uploaded file URL, extracts the financial data using Gemini, and maps it to the mandatory fields defined in CostTemplate for that provider type. Any extracted costs that do not match the template should be saved as isCustom: true.

Workflow Endpoints: * POST /costs/submit: Saves the parsed ProviderCostSheet and CostItems as PENDING. Triggers a system notification and an email (via the existing SendGrid service) to the GoStork Admin.

POST /costs/approve/:id: Changes status to APPROVED. Archives the previous approved version to maintain full history.

POST /costs/reject/:id: Changes status to REJECTED. Accepts a feedback string. Triggers a system notification and SendGrid email back to the provider with the admin's feedback.

3. Frontend (React/Vite & shadcn/ui):

UI Integration: Insert the new <ProviderCostsTab /> component immediately after the "Company" tab in both the provider settings page AND the admin's admin-provider-edit-page.tsx.

Provider View: * Display an upload zone for PDF/Excel.

Upon upload and AI parsing, render a dynamic form of the costs.

Visually distinguish (e.g., a subtle yellow background or distinct badge) fields that are isCustom: true (non-GoStork fields extracted by AI).

Allow the provider to manually edit values, add new rows, or delete custom rows.

Validation: Prevent submission unless all fields marked isMandatory in the GoStork template have a valid numerical value or range.

Admin View (admin-provider-edit-page.tsx):

Admin must be able to manage the global CostTemplate (add/edit/remove mandatory fields per provider type).

When reviewing a PENDING ProviderCostSheet, display a "Diff View" comparing the currently APPROVED sheet against the PENDING sheet so the admin can easily spot changes.

Highlight isCustom fields.

Provide an "Approve" button and a "Reject" button (which opens a modal to input mandatory feedback).

Strict Component Rule: Use the exact same <ProviderCostsTab /> component for all provider types (IVF, Surrogacy, Egg Donor) by passing the providerType as a prop to fetch the correct template, rather than hardcoding separate views.
